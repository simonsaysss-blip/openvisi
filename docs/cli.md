# OpenVisi CLI

OpenVisi is an open-source measurement layer for AI Visibility.

AI Visibility is the measurable presence, accuracy, citation quality, and competitive position of an entity across AI-generated answers.

The CLI supports config initialization, deterministic dry-run scan planning, and static crawl artifact generation. It does not call LLM providers, compute answer-level metrics, or write completed scan results.

## Create a Config

```bash
openvisi init
```

This creates:

```text
openvisi.config.json
```

The command does not overwrite an existing config unless `--force` is passed:

```bash
openvisi init --force
```

The starter config contains no API keys.

## Config Shape

```json
{
  "brandName": "OpenVisi",
  "domain": "openvisi.dev",
  "category": "AI Visibility diagnostics",
  "competitors": [
    {
      "name": "Example Competitor",
      "domain": "example.com",
      "aliases": ["Example"]
    }
  ],
  "providers": [
    {
      "provider": "mock",
      "model": "mock-v0",
      "enabled": true
    }
  ],
  "outputDir": "openvisi-report",
  "includeExperimentalMetrics": false,
  "promptPackInput": {
    "audience": "B2B SaaS teams",
    "problems": [
      "measuring AI-generated answer visibility",
      "understanding how AI describes a brand"
    ],
    "integrations": ["Google Search Console", "Slack"]
  }
}
```

If `promptPack` is omitted, the CLI generates a deterministic default prompt pack from `promptPackInput`, `brandName`, `domain`, `category`, and `competitors`.

## Dry-run Scan Planning

```bash
openvisi scan --dry-run --provider mock --output openvisi-report
```

Options:

- `--config <path>`: read a specific OpenVisi config file
- `--output <path>`: override `outputDir`
- `--provider <provider>`: use one enabled provider in the normalized config
- `--include-experimental-metrics`: set `includeExperimentalMetrics` to `true`
- `--dry-run`: write a reproducible scan plan without crawling or provider calls

## Dry-run Output Files

```text
openvisi-report/
  scan-plan.json
  prompt-pack.json
  config.normalized.json
  artifact-manifest.json
  warnings.json
```

`scan-plan.json` summarizes the planned scan:

- brand name
- domain
- category
- enabled providers
- prompt count
- prompt intents
- competitor count
- generated timestamp
- next implementation stage

`prompt-pack.json` contains the generated or provided `PromptSpec[]`.

`config.normalized.json` contains the materialized `OpenVisiScanConfig`.

`warnings.json` explains dry-run limitations.

`artifact-manifest.json` records the dry-run artifacts generated by the command. Dry-run does not write `report-references.json` because no crawl evidence exists.

## What Stage 1C Does Not Do

- It does not crawl pages.
- It does not call LLM providers.
- It does not compute metrics.
- It does not write `scan-result.json`.
- It does not create fake answers, fake citations, or fake crawled page snapshots.

## Static Crawl Artifacts

```bash
openvisi crawl --config openvisi.config.json --output openvisi-report --render-mode static
```

Options:

- `--config <path>`: read a specific OpenVisi config file
- `--output <path>`: override the crawler artifact output directory
- `--url <url>`: override the config domain with a seed URL
- `--max-pages <number>`: pass a static crawl page limit
- `--render-mode static`: use the current static crawler

`--render-mode headless` is accepted only to produce a clear warning. Headless crawling is not implemented yet, and the CLI falls back to static crawling.

## Crawl Output Files

```text
openvisi-report/
  crawled-pages.json
  crawler-summary.json
  structure-trust-inputs.json
  artifact-manifest.json
  report-references.json
  warnings.json
```

`crawled-pages.json` is a canonical `CrawledPageSnapshot[]` artifact. It contains static crawler output normalized into OpenVisi's scan contract, including available machine-readable trust and AI-readable structure diagnostics.

`crawler-summary.json` includes aggregate diagnostic counts:

- pages with JSON-LD
- pages with Organization schema
- pages with Product or SoftwareApplication schema
- pages with FAQ schema
- pages with author metadata
- pages with last-modified metadata
- pages with canonical URLs
- HTTPS pages
- pages with clear H1 structure
- pages with docs-like structure
- pages with FAQ section signals
- pages with comparison signals

`structure-trust-inputs.json` summarizes crawler-derived AI-readable Structure, Machine-readable Trust, AI Citation Signal inputs, preferred source domains, source gap candidates, and limitations. It is not `metrics.json` and does not include `aiVisibilityScore`.

`warnings.json` explains static-only limitations and whether diagnostics may be incomplete because source data was missing.

`artifact-manifest.json` records the generated crawl artifacts.

`report-references.json` maps crawler artifacts to future report sections, including AI-readable Structure, Machine-readable Trust, AI Citation Signals, and Source Gaps.

## Mock Evaluation Artifacts

```bash
npx openvisi eval --provider mock --output openvisi-report
```

Options:

- `--config <path>`: read a specific OpenVisi config file
- `--output <path>`: override the evaluation artifact output directory
- `--provider <provider>`: only `mock` is supported in Stage 3B
- `--model <model>`: mock model label, defaulting to `mock-v0`
- `--include-experimental-metrics`: set `includeExperimentalMetrics` in the normalized config

Unsupported providers fail with:

```text
Real provider adapters are not implemented in Stage 3B. Use --provider mock.
```

Evaluation output files:

```text
openvisi-report/
  config.normalized.json
  prompt-pack.json
  answers.json
  answer-signal-inputs.json
  artifact-manifest.json
  warnings.json
```

`answers.json` contains deterministic mock `LLMAnswer[]` records. It is not real LLM output, not `metrics.json`, and not `scan-result.json`.

`answer-signal-inputs.json` summarizes transparent signals derived from `answers.json`. It is not final metrics and does not include `aiVisibilityScore`.

## Compose Measurement Inputs

```bash
npx openvisi inputs compose \
  --crawl-output openvisi-crawl \
  --eval-output openvisi-eval \
  --output openvisi-measurement
```

Defaults:

- `--crawl-output openvisi-crawl`
- `--eval-output openvisi-eval`
- `--output openvisi-measurement`

The command expects:

- a valid `static-crawl` artifact bundle containing `structure-trust-inputs.json`
- a valid `evaluation` artifact bundle containing `answer-signal-inputs.json`

Output files:

```text
openvisi-measurement/
  measurement-inputs.json
  artifact-manifest.json
  warnings.json
```

`measurement-inputs.json` embeds parsed structure/trust inputs and answer signal inputs for a future metrics composer. It is not `metrics.json`, does not include `aiVisibilityScore`, and does not imply a completed AI Visibility scan.

The command does not copy raw upstream artifacts such as `crawled-pages.json`, `answers.json`, `structure-trust-inputs.json`, or `answer-signal-inputs.json`.

## Compose Metrics Draft

```bash
npx openvisi metrics draft \
  --measurement-output openvisi-measurement \
  --output openvisi-metrics-draft
```

Defaults:

- `--measurement-output openvisi-measurement`
- `--output openvisi-metrics-draft`

The command expects a valid `measurement-inputs` artifact bundle containing `measurement-inputs.json`.

Output files:

```text
openvisi-metrics-draft/
  metrics-draft.json
  artifact-manifest.json
  warnings.json
```

`metrics-draft.json` contains explainable draft metric values with `derivedFrom` and `explanation` fields. It is not `metrics.json`, does not include final `aiVisibilityScore`, and does not imply a completed AI Visibility scan.

The command does not write `metrics.json`, `scan-result.json`, reports, citations, or raw upstream source artifacts.

## Guard Final Metrics Generation

```bash
npx openvisi metrics guard \
  --metrics-review-output openvisi-metrics-review \
  --output openvisi-metrics-finalization
```

Defaults:

- `--metrics-review-output openvisi-metrics-review`
- `--output openvisi-metrics-finalization`

The command expects a valid `metrics-review` artifact bundle containing `metrics-review.json`.

Output files:

```text
openvisi-metrics-finalization/
  metrics-finalization.json
  artifact-manifest.json
  warnings.json
```

`metrics-finalization.json` decides whether future final metrics generation is allowed. It is not `metrics.json`, does not compute final `aiVisibilityScore`, and blocks finalization under mock evaluator evidence.

The command does not write `metrics.json`, `scan-result.json`, reports, citations, or raw upstream source artifacts.

## Review Metrics Draft

```bash
npx openvisi metrics review \
  --metrics-draft-output openvisi-metrics-draft \
  --output openvisi-metrics-review
```

Defaults:

- `--metrics-draft-output openvisi-metrics-draft`
- `--output openvisi-metrics-review`

The command expects a valid `metrics-draft` artifact bundle containing `metrics-draft.json`.

Output files:

```text
openvisi-metrics-review/
  metrics-review.json
  artifact-manifest.json
  warnings.json
```

`metrics-review.json` gates future final metrics composition. It is not `metrics.json`, does not compute final `aiVisibilityScore`, and blocks finalization when evaluator evidence is mock.

The command does not write `metrics.json`, `scan-result.json`, reports, citations, or raw upstream source artifacts.

## Artifact Bundle Inspection

```bash
openvisi artifacts inspect --output openvisi-report --stage dry-run
openvisi artifacts inspect --output openvisi-report --stage static-crawl
openvisi artifacts inspect --output openvisi-report --stage evaluation
openvisi artifacts inspect --output openvisi-measurement --stage measurement-inputs
openvisi artifacts inspect --output openvisi-metrics-draft --stage metrics-draft
openvisi artifacts inspect --output openvisi-metrics-review --stage metrics-review
openvisi artifacts inspect --output openvisi-metrics-finalization --stage metrics-finalization
```

Options:

- `--output <path>`: artifact bundle directory, defaulting to `openvisi-report`
- `--stage <stage>`: expected bundle stage: `dry-run`, `static-crawl`, `evaluation`, `measurement-inputs`, `metrics-draft`, `metrics-review`, `metrics-finalization`, `full-scan`, `report`, or `unknown`

The command reads `artifact-manifest.json`, checks whether referenced artifact files exist, and prints a concise validation summary. It exits non-zero when error-level validation issues are found.

The artifact contracts live in `packages/core`; the CLI only provides filesystem reading, writing, and inspection.

## What Stage 4C Does Not Do

- It does not use Playwright or Puppeteer.
- It does not render JavaScript-heavy pages with a browser.
- It does not call real LLM providers.
- It does not create `metrics.json`, `citations.json`, or `scan-result.json`.
- It does not generate real answers, citations, metrics, or crawled page content.
- It does not generate `report.md` or `report.html` from an artifact bundle.
