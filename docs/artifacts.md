# OpenVisi Artifact Bundles

OpenVisi output directories are structured artifact bundles.

Stage 2C added a manifest layer so later report, evaluator, metrics, storage, and dashboard stages can discover which artifacts exist without guessing filenames.

Stage 2D moves the shared artifact contracts into `packages/core` so downstream packages can consume artifact bundles without depending on the CLI package. The CLI remains responsible for filesystem reads, writes, and command-line inspection.

## artifact-manifest.json

`artifact-manifest.json` records generated artifacts in the current output directory:

```json
{
  "schemaVersion": "0.1",
  "project": "OpenVisi",
  "generatedAt": "2026-05-26T00:00:00.000Z",
  "stage": "static-crawl",
  "artifacts": [],
  "warnings": []
}
```

The manifest only references files produced by the current command stage. It does not add entries for metrics, answers, citations, or completed scan results unless those files actually exist in a future stage.

## report-references.json

`report-references.json` maps generated artifacts to future report sections.

For static crawl output:

- `crawled-pages.json` is evidence for AI-readable Structure.
- `crawler-summary.json` is evidence for Machine-readable Trust.
- `warnings.json` is evidence for Source Gaps.

These references do not mean a full report has been generated. They make evidence artifacts discoverable for later report generation.

## structure-trust-inputs.json

Stage 2F adds `structure-trust-inputs.json` as a crawler-derived input artifact.

It is generated from:

- `crawled-pages.json`
- `crawler-summary.json`

It contains:

- AI-readable Structure signal counts
- Machine-readable Trust signal counts
- AI Citation Signal input booleans
- preferred source domains
- source gap candidates
- limitations

It intentionally excludes:

- `aiVisibilityScore`
- computed OpenVisi metrics
- LLM-generated answers
- citations
- completed `scan-result.json`
- Markdown or HTML reports

The artifact exists so future evaluator, metrics, report, storage, and dashboard modules can consume crawler-derived evidence through a stable contract before full scan results exist.

## answers.json

`answers.json` is an evaluation-stage artifact type prepared by Stage 3A and generated by `openvisi eval` in Stage 3B.

It contains an `AnswersArtifact` wrapper with:

- source references to `prompt-pack.json` and `config.normalized.json`
- provider name
- model name
- `LLMAnswer[]`

It is not produced by dry-run or static crawl commands.

It is not `metrics.json`.

It is not `scan-result.json`.

It does not include `aiVisibilityScore`.

## answer-signal-inputs.json

`answer-signal-inputs.json` is an evaluator-derived input artifact generated by `openvisi eval`.

It is generated from:

- `answers.json`
- `prompt-pack.json`
- `config.normalized.json`

It contains:

- answer presence signal counts
- entity clarity signal counts
- citation coverage input counts
- competitor displacement input counts
- narrative accuracy input markers
- prompt-level signal results
- limitations

It intentionally excludes:

- `aiVisibilityScore`
- computed OpenVisi metrics
- separate `citations.json`
- completed `scan-result.json`
- Markdown or HTML reports

It is designed as an input to future metrics and reporting stages, not as final metrics.

## Evaluation Artifact Bundle

The evaluation stage writes:

```text
openvisi-report/
  config.normalized.json
  prompt-pack.json
  answers.json
  answer-signal-inputs.json
  artifact-manifest.json
  warnings.json
```

The evaluation bundle intentionally excludes `metrics.json`, `citations.json`, `scan-result.json`, reports, crawled pages, and `structure-trust-inputs.json`.

`answers.json` contains `LLMAnswer[]` and does not imply metrics have been computed.

`answer-signal-inputs.json` contains evaluator-derived input signals and also does not imply metrics have been computed.

## measurement-inputs.json

Stage 3D adds `measurement-inputs.json` as a composed downstream input artifact.

It is generated from two upstream artifact bundles:

- `structure-trust-inputs.json` from a `static-crawl` bundle
- `answer-signal-inputs.json` from an `evaluation` bundle

It contains:

- source bundle references
- source artifact references
- input completeness flags
- evidence summary counts
- embedded structure/trust input signals
- embedded answer signal inputs
- limitations

It intentionally excludes:

- `aiVisibilityScore`
- computed OpenVisi metrics
- raw `answers.json`
- raw `crawled-pages.json`
- separate citation artifacts
- completed `scan-result.json`
- Markdown or HTML reports

`measurement-inputs.json` is the final input bundle before future metrics composition. It is not `metrics.json`.

## Measurement Inputs Artifact Bundle

The measurement-inputs stage writes:

```text
openvisi-measurement/
  measurement-inputs.json
  artifact-manifest.json
  warnings.json
```

The bundle intentionally excludes upstream payload files such as `answers.json`, `crawled-pages.json`, `structure-trust-inputs.json`, and `answer-signal-inputs.json`.

## metrics-draft.json

Stage 4A adds `metrics-draft.json` as an explainable draft metrics artifact.

It is generated from:

- `measurement-inputs.json`

It contains:

- draft metric values
- availability flags
- `derivedFrom` source labels
- explanations for every draft formula
- excluded metrics
- completeness flags
- limitations

It intentionally excludes:

- final `aiVisibilityScore`
- `metrics.json`
- completed `scan-result.json`
- separate citation artifacts
- Markdown or HTML reports

`metrics-draft.json` is for review and debugging before final metrics composition. It is not final measurement output.

## Metrics Draft Artifact Bundle

The metrics-draft stage writes:

```text
openvisi-metrics-draft/
  metrics-draft.json
  artifact-manifest.json
  warnings.json
```

The bundle intentionally excludes `metrics.json`, `scan-result.json`, raw `measurement-inputs.json`, raw answers, crawl artifacts, citations, and reports.

## metrics-review.json

Stage 4B adds `metrics-review.json` as a review gate artifact for `metrics-draft.json`.

It is generated from:

- `metrics-draft.json`

It contains:

- readiness flags
- per-metric review decisions
- blocking reasons
- recommended next actions
- limitations

It intentionally excludes:

- final `aiVisibilityScore`
- `metrics.json`
- completed `scan-result.json`
- separate citation artifacts
- Markdown or HTML reports

`metrics-review.json` prevents premature final scoring, especially when evaluator evidence is mock.

## Metrics Review Artifact Bundle

The metrics-review stage writes:

```text
openvisi-metrics-review/
  metrics-review.json
  artifact-manifest.json
  warnings.json
```

The bundle intentionally excludes `metrics.json`, `scan-result.json`, raw `metrics-draft.json`, raw source artifacts, citations, and reports.

## metrics-finalization.json

Stage 4C adds `metrics-finalization.json` as a finalization guard artifact.

It is generated from:

- `metrics-review.json`

It contains:

- finalization status
- source artifact reference
- readiness flags
- generation permission decisions
- blocking reasons
- required actions before finalization
- limitations

It intentionally excludes:

- final `aiVisibilityScore`
- `metrics.json`
- completed `scan-result.json`
- separate citation artifacts
- Markdown or HTML reports

`metrics-finalization.json` decides whether a future stage is allowed to generate final metrics. Under mock evaluator evidence, finalization is blocked.

## Metrics Finalization Artifact Bundle

The metrics-finalization stage writes:

```text
openvisi-metrics-finalization/
  metrics-finalization.json
  artifact-manifest.json
  warnings.json
```

The bundle intentionally excludes `metrics.json`, `scan-result.json`, raw `metrics-review.json`, raw source artifacts, citations, and reports.

## Artifact Inspection

Use the CLI to inspect an output directory:

```bash
openvisi artifacts inspect --output openvisi-report --stage static-crawl
```

The command reads `artifact-manifest.json`, verifies referenced files exist, optionally validates the expected stage, and exits non-zero when error-level validation issues are found.

`artifact-manifest.json` is the entry point for downstream modules. The CLI does not load large artifact payloads during inspection; it validates bundle structure and references only.

## Golden Fixture Bundles

Golden fixture bundles live under:

```text
test/fixtures/artifact-bundles/
  dry-run/
  static-crawl/
  evaluation/
  measurement-inputs/
  metrics-draft/
  metrics-review/
  metrics-finalization/
  invalid-missing-file/
  invalid-absolute-path/
```

The dry-run fixture represents a config and prompt planning bundle. It includes `artifact-manifest.json`, `config.normalized.json`, `prompt-pack.json`, `scan-plan.json`, and `warnings.json`.

The static-crawl fixture represents canonical static crawl artifact output. It includes `artifact-manifest.json`, `crawled-pages.json`, `crawler-summary.json`, `structure-trust-inputs.json`, `report-references.json`, and `warnings.json`.

Fixtures intentionally exclude artifacts that do not belong to their stage. For example, dry-run and static-crawl fixtures exclude answers, metrics, citations, completed `scan-result.json`, Markdown reports, and HTML reports. Evaluation fixtures include `answers.json` but exclude final metrics and completed scan results. They exist to prevent artifact contract drift before evaluator, metrics, report, storage, and dashboard modules consume these bundles.

The measurement-inputs fixture represents the composed input bundle consumed by a future metrics composer. It intentionally excludes raw upstream payload artifacts and final metrics.

The metrics-draft fixture represents transparent draft metrics derived from measurement inputs. It intentionally excludes final metrics, scan results, reports, and source payload files.

The metrics-review fixture represents the review gate over draft metrics. It intentionally excludes final metrics, scan results, reports, and source payload files.

The metrics-finalization fixture represents the finalization guard over metrics review output. It intentionally excludes final metrics, scan results, reports, and source payload files.

Downstream modules should validate artifact bundles before consuming them.

## Current Stage Limits

Stage 4C does not compute final metrics.

Stage 4C does not call real LLM providers, generate separate citation artifacts, or create completed `scan-result.json` files.

Stage 4C does not generate full reports from the artifact bundle. Existing report behavior remains separate.
