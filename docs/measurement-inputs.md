# Measurement Inputs

Stage 3D introduces `measurement-inputs.json`.

This artifact combines the two downstream input artifacts produced before metrics composition:

- `structure-trust-inputs.json` from a `static-crawl` artifact bundle
- `answer-signal-inputs.json` from an `evaluation` artifact bundle

It is an input bundle only.

It is not `metrics.json`.

It does not compute `aiVisibilityScore`.

It does not imply a completed AI Visibility scan.

## Command

```bash
npx openvisi inputs compose \
  --crawl-output openvisi-crawl \
  --eval-output openvisi-eval \
  --output openvisi-measurement
```

The command writes:

```text
openvisi-measurement/
  measurement-inputs.json
  artifact-manifest.json
  warnings.json
```

It does not copy raw upstream payload artifacts such as `crawled-pages.json`, `answers.json`, `structure-trust-inputs.json`, or `answer-signal-inputs.json`.

## What It Contains

`measurement-inputs.json` contains:

- source bundle references
- source artifact references
- input completeness flags
- evidence summary counts
- embedded structure/trust input signals
- embedded answer signal inputs
- limitations

The artifact is safe for a future metrics composer to consume through `@openvisi/core` contracts.

## Limitations

This stage does not compute final metrics.

It does not include `aiVisibilityScore`.

It does not generate `scan-result.json`.

If the evaluation bundle was generated by the mock provider, the composed inputs are deterministic mock evidence and should not be treated as real LLM behavior.
